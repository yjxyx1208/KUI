二、等待心理学

为什么用户对 AI 的延迟容忍度反常地高？

延迟悖论：在 AI 场景下，即时响应反而让用户觉得「太浅了」，适度延迟 + 可见的思考过程 = 更高的信任感和智能感。这彻底颠覆了传统 UX 的「越快越好」准则。

Jakob Nielsen 的响应时间三阈值（AI 时代更新版）

阈值传统软件AI 产品
0.1s 即时感 同，但 AI 场景罕见
1s 思维流不中断 用户开始期待「思考」反馈
3s 需要 spinner TTFT>3s 流失率 70%+（国内数据）
10s 注意力丧失 必须提供 ETA + 允许离开

学术发现：刻意延迟增加信任

ShareOK 研究（Oklahoma State University）证实：动态延迟响应（模拟人类思考节奏）显著增加用户认知信任，但对情感信任无显著影响。

换句话说：用户不需要 AI 假装是人，但需要 AI 表现出「在认真对待我的问题」。

三、等待体验设计策略全景

10 种被验证有效的策略，从标配到前沿

各产品等待设计策略采用矩阵

📡

流式输出 (Streaming)

通过 SSE/WebSocket 逐 token 渲染，将「等待结果」转化为「见证创作」。搭配打字光标动画，是所有产品的基础设施。

全部产品 效果：消除「卡住了吗？」焦虑
🧠

思考过程展示 (Stream of Thought)

展示 AI 的推理链、搜索过程、工具调用。最强的信任建立手段，也是当前差异化主战场。

ChatGPT Claude DeepSeek Perplexity 效果：信任 + 智能感显著提升
🔍

阶段指示器 (Stage Indicators)

「正在搜索 3 个来源...」「正在分析数据...」「正在生成报告...」— 用状态替代时间。

Perplexity ChatGPT Kimi 效果：降低不确定性焦虑
⏹

可中断控制 (Interruptibility)

允许用户在生成过程中停止、编辑、重定向。恢复用户控制感，减轻「只能干等」的无力感。

ChatGPT Claude 被低估的机会点
💡

语义骨架 / 任务预告

不是展示灰色占位块，而是预告「这次回答将包含：3个要点 + 1个表格 + 2个来源链接」。

前沿探索中 效果：设定预期 + 降低焦虑
四、竞品等待设计拆解

各家的设计哲学和具体实现

ChatGPT — 「忙碌但透明」

TTFT：GPT-4o ~300ms A，o3 推理模式 2-10s+ A

等待设计：thinking 过程可折叠展示，搜索时展示正在访问的 URL，工具调用时展示工具名称和参数。深度研究模式下有详细的多阶段进度。

设计哲学：让用户看到 AI 在做什么，但不强迫看完。

Claude — 「极简信任」

TTFT：Haiku ~600ms A，Sonnet ~1-2s A，Opus ~1-3s B

等待设计：极简打字光标 + 流式输出。Extended thinking 模式下展示思考过程（可展开查看完整推理链）。Claude Code 展示工具调用进度。

设计哲学：少即是多，信任来自输出质量而非过程炫技。

Gemini — 「快是第一性原理」

TTFT：Flash ~730ms A，Pro 标准速度。官方称 Flash Lite 比 Flash 再快 2.5x B

等待设计：默认使用 Flash（速度优先），复杂任务自动切换 Pro。thinking 模式有进度展示。

设计哲学：用模型分层解决延迟问题，而非靠 UI 掩盖。

Perplexity — 「搜索可视化」

TTFT：取决于搜索链路，通常 2-5s B 估算

等待设计：最丰富的过程展示 — 实时展示搜索查询、正在阅读的来源、分析进度。结果优先（先出答案摘要，再展开详情和来源）。

设计哲学：让搜索过程本身成为内容，等待 = 学习。

豆包 — 「速度即体验」

TTFT：语音场景首包延迟 ~182ms B 厂商数据，文本 TTFT 缺乏独立测试数据

等待设计：依托字节跳动的推理基础设施，以极低延迟直接减少等待感知。具体 UI 实现细节需实机验证。

设计哲学：速度本身就是最好的 UX — 如果够快，就不需要复杂的等待设计。

Kimi & DeepSeek

Kimi：moonshot v1 8k TTFT ~0.2s B 单一来源。长文本处理见长，Mooncake 架构提升推理吞吐（官方称 525%↑）。具体等待 UI 实现需实机验证。

DeepSeek：V3 简单任务 ~0.2s B，R1 推理模式延迟变动大。R1 可暴露完整思维链（开源社区及官方 API 文档可确认）。开源部署差异导致延迟数据不具代表性。

五、灵光App 落地建议

基于竞品分析和用户心理学研究，分优先级落地

核心策略：不追求单一的「快」，而是为不同场景设计「有意义的等待」。国内用户耐心阈值低于海外，等待设计应偏「丰富」而非「极简」。
P0
流式输出 + 光标动画

基础体验保障。SSE/WebSocket 逐 token 渲染 + 打字光标动画。所有场景必须。

P0
阶段指示器

推理/搜索类任务必须。「正在分析...」「正在搜索...」用状态替代空白。实施成本低。

P1
思考过程展示（可折叠）

深度推理任务的差异化手段。默认收起，好奇的用户可展开。对标 ChatGPT thinking / DeepSeek 思维链。

P1
模型分层等待 UI

如果灵光使用多个模型，快模型用简洁等待，慢模型用丰富等待。避免「同一个 APP 有时快有时慢」的体感割裂。

P2
语义骨架 / 任务预告

复杂生成任务预告内容结构：「本次回答将包含 3 个要点 + 1 个示例」。降低不确定性。

P2
细粒度中断控制

超越简单的「Stop」按钮。提供「到这里就好」「换个角度」等控制。长文本生成场景尤其需要。

P3
刻意微延迟

简单问答场景，快模型响应太快反而显得「不认真」。可在 TTFT<200ms 时加入 200-500ms 的模拟思考延迟。

六、关键结论

1. TTFT 竞争格局：速度不是唯一战场

豆包/Groq 在绝对速度上领先，但 Claude/ChatGPT 证明了「有意义的等待」可以转化为信任优势。灵光不必卷速度本身。

2. 设计哲学选择：「丰富型」更适合灵光

国内用户 TTFT>3s 流失率超 70%。灵光作为国内产品，应选择「丰富但有节奏」的等待设计，而非 Claude 式极简。参考 Perplexity 的过程可视化思路。

3. 下一个蓝海：可中断性设计

目前所有产品的中断控制都很粗糙（只有 Stop）。细粒度的中断控制（编辑方向、部分采纳、checkpoint）是差异化的未开发空间。
