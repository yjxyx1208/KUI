基于 TTFT 竞品研究洞察，P0-P2 落地建议的交互原型

P0 流式输出 + 光标动画

基础体验保障。逐 token 渲染，光标动画暗示「正在输出」，消除「卡住了？」焦虑。

设计要点：光标在最后一个字符后闪烁，用户视线自然跟随文字增长。流式输出让等待从「干等结果」变成「围观创作」。

实现建议

  • SSE (Server-Sent Events) 或 WebSocket 传输
  • 光标宽度 2px，闪烁周期 500-600ms，颜色跟随品牌色
  • token 到达间隔不均匀时做 buffer 平滑，避免视觉卡顿
  • 首个 token 到达前展示光标 + 轻微 shimmer，暗示「准备中」

P0 阶段指示器

推理/搜索类任务必须。用「状态」替代空白，让用户知道 AI 在做什么，而非只是「在转圈」。

设计要点：阶段从上到下推进，已完成的步骤变绿打勾，当前步骤琥珀色脉冲动画。不用展示具体耗时，状态本身就在消解焦虑。

实现建议

  • 阶段数量控制在 3-5 个，太多反而增加认知负担
  • 阶段文案用具体动词：「搜索 3 个来源」比「加载中」有价值
  • 后端需要在每个阶段转换时推送状态事件
  • 最后一个阶段完成后无缝切换到流式输出

P1 思考过程展示（可折叠）

深度推理任务的差异化手段。默认收起（不打扰），好奇的用户可展开看 AI 在想什么。

  • 正式回答开始后思考块自动折叠，避免占据太多视口
  • 仅在深度推理模型/模式下展示，快速问答不需要

P1 模型分层等待 UI

快模型用简洁等待，慢模型用丰富等待。避免「同一个 APP 有时快有时慢」的体感割裂。

实现建议

  • Header 区域用颜色标签区分当前模型/模式（绿色=快，紫色=深度）
  • 快模型：只用光标动画，不展示阶段，不展示思考过程
  • 慢模型：阶段指示器 + 可折叠思考过程 + 耗时显示
  • 自动切换模型时需要 UI 过渡提示：「切换到深度模式以获得更好的结果」

P2 语义骨架 / 任务预告

不是灰色占位块，而是预告「这次回答将包含什么」。用内容预期替代时间焦虑。

设计要点：在正式内容到达前，先展示回答的「目录」。shimmer 动画表示正在加载，各项逐步替换为实际内容。

实现建议

  • 需要后端在生成前快速预估回答结构（可用轻量模型预跑）
  • 预告项逐一被实际内容替换，产生「填充」的满足感
  • 仅在复杂/长文本任务触发，简单问答不需要
  • 如果预估不准确，宁可少展示，不要展示后删除

P2 细粒度中断控制

超越简单的「Stop」。让用户在生成过程中有更多控制权：到此为止、换个方向、更简洁。

量子计算经过近四十年的发展，正从实验室走向工程化阶段。2025-2026 年，IBM、Google、IonQ 等公司相继突破了...

二、商业化路径
目前主要有三条路径：量子即服务（QaaS）、混合量子-经典计算、以及垂直行业解决方案...
输入消息...
设计要点：按钮在生成过程中出现，结束后消失。「到这里就好」保留已生成内容；「换个角度」保留内容但切换论述方向；「更简洁」让 AI 压缩后续输出。

实现建议

  • 按钮组在首段生成完成后出现（太早出现会干扰阅读）
  • 「到这里就好」= 保留已输出 + 追加简短收尾
  • 「换个角度」= 停止当前 + 发送 system 指令 + 续写
  • 「更简洁」= 修改 generation 参数（降低 max_tokens、提高 temperature）
  • 按钮不超过 4 个，多了就是选择负担
